# 并发指纹与会话隔离规约

本规约解决 AI 在逆向任务中最容易犯的第三类错误：还没验证稳定性就上并发，把 session/cookie/token/cache 混用，把指纹问题当成一个词就能解释的问题。

本仓库处理 Web/H5 的指纹、WAF、风控时，必须把每个字段、cookie、profile、token、失败码和恢复动作都挂到具体 capture/run/session 上；不能把旧值、脏 profile 或一次临时恢复当成稳定工程结论。

## 并发前置条件

进入并发前必须满足：

- 单请求可复现，且保存请求/响应证据。
- 连续请求基线已跑，记录 N 次成功率和失败码。
- cookie jar、localStorage、sessionStorage、token cache 的归属清楚。
- 已区分业务错误、参数错误、WAF/风控错误、频控错误。
- 已定义停止条件：失败率、403/429/503 比例、连续失败次数、最大重试次数。

## 并发阶梯

并发只能按阶梯推进：

```text
1 worker baseline
2 workers isolation check
5 workers stability check
10 workers only if previous stage is stable
```

每一阶都记录：

- worker 数。
- 每个 worker 的 session scope。
- 请求总数、成功数、失败数。
- 403/429/503 比例。
- 平均耗时和 P95。
- token/cookie 刷新次数。
- 是否出现跨 worker 污染。

## 隔离规则

并发 worker 之间默认隔离：

- browser context 不共享。
- cookie jar 不共享。
- localStorage/sessionStorage 不共享。
- 风控 token cache 不共享，除非证据证明 cache key 完整且可复用。
- 账号态不共享，除非任务明确允许并记录授权边界。

cache key 至少包含：

```text
domain + host + market + locale + currency + stage + auth_state + UA/client hints + browser profile scope + session scope
```

遇到 WAF/指纹时，额外记录：

- 保护供应商迹象：Imperva / Akamai / Cloudflare / 其他。
- 当前 fingerprint/cookie/token 是在哪个 session scope 下生成的。
- protected business API 是否接受，而不只是 challenge/token endpoint 成功。
- 重启浏览器、清空 profile、切换 market 后是否仍稳定。

## 恢复与稳定性策略

可记录和验证的工程动作：

- 降速、退避、jitter。
- 停止并发，回到单线程复现。
- 重新录制真实浏览器 HAR/fixtures。
- 重建隔离的 browser context。
- retire 可疑 session,不把 doubtful session 继续复用。
- 输出 risk-control concurrency ledger 和 acceptance report。
- 标记失败为 known-failure，等待人工或授权环境。

不合格做法：

- 未记录授权范围、会话归属和 capture_id 就切换账号、代理、profile 或设备态。
- 复制旧 clearance cookie、risk token、设备 ID 或 fingerprint 值,却声称是本轮 fresh evidence。
- 没有说明 fingerprint 字段来源、TTL、绑定关系和业务 API 接受度。
- 用无限重试掩盖失败。
- 把风控触发后的临时恢复说成稳定交付。

## 交付要求

凡是声称“批量”“并发”“高并发”“稳定运行”，必须交付：

- 并发阶梯记录。
- session/cache 隔离说明。
- 失败率和恢复策略。
- 未覆盖的 market/stage/session。
- kill switch 或停止条件。
- `tools/web_h5_acceptance_report.py validate --require-complete` 通过的 acceptance report。
- protected business API backend acceptance,不能只看 challenge/token endpoint。
- metrics: success_browserless_verified、concurrency_verified、flaky_count、blocked_by_protection。

未跑并发时，只能说“单线程已验证”或“连续请求已验证”，不能说“支持并发”。

## 风控工程边界

本规约里的风控处理是并发安全和证据治理,不是绕过能力。允许记录和实现隔离、降速、退避、jitter、session retirement、kill switch、fresh replay 和人工复核。禁止沉淀 WAF/CAPTCHA 绕过、webdriver 隐藏、指纹伪造、代理轮换、clearance cookie 或旧风险 token 复用。
