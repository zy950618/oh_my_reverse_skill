# 网页逆向标准规划

> **触发场景**：用户说"网页逆向 XXX 网站"、"逆向 XXX"、"实现 XXX 站点纯接口"、"FastAPI 接口测试交付"、"把 XXX 接入 314"、"做一下 XXX 的爬虫"、"破解 XXX 的 sign/token"等。
>
> **本文不是 SKILL.md，是 meta 规划文档**。Claude 读这份文档来理解"如何从一句话需求出阶段化规划"。具体执行交给 `1-业务流程层/website-314-api-delivery` 及其子 skill。

一句话目标: 在授权范围内实现全站关键接口处理、加密/签名逆向和关键接口测试交付; WAF/验证码/风控状态只能按真实后端接受证据声明能力,不能把未验证 challenge、403 页面或 adapter-only 结果写成成功。

---

## 0. 触发后的第一动作（不要省）

收到用户的"网页逆向 XXX"请求时，**先输出一份阶段化规划给用户过目，再开干**。规划格式固定如下，确保信息密度：

```
目标：<URL + 业务目标>
分类：<纯HTTP / JS加密 / 补环境 / WAF / 接口化沉淀 / FastAPI接口测试交付 / 本地基础框架接入>
难度评估：<低/中/高> + 关键不确定点 3 个
阶段化计划：
  1. 侦察    skill: <name>    产出: <什么>
  2. 入口    skill: <name>    产出: <什么>
  3. 还原    skill: <name>    产出: <什么>
  4. 复现    skill: <name>    产出: <什么>
  5. 沉淀    skill: <name>    产出: <什么>
  6. 一致性  skill: <name>    产出: <什么>
  7. 收尾    skill: governance    产出: cleanup ledger + encryption algorithm graph
逆向经验库检查：<是否已有 逆向工程经验库/domains/<domain>/reverse-memory.md，已有则先读完>
站点经验库检查：<是否已有 站点经验库/<domain>/，已有则先读完 known-failures.md>
WAF 预判：<是否存在 reese84/incapsula/akamai/cloudflare 痕迹>
风险与红线：<是否涉及真实扣款 / 是否涉及登录态 / 是否需要授权>
证据口径：<哪些结论必须给 URL/HAR/调用栈/JSON Pointer/命令输出>
范围账本：<domain/market/locale/currency/stage/auth_state/target_api>
并发会话边界：<是否涉及批量/并发/指纹/session/cache，若涉及如何隔离>
图谱更新：<本次会新增/修改哪些节点和边,落到 knowledge-graph.md>
影响回归：<direct_impact / downstream_impact / required_regression,落到 impact-regression.md>
证据新鲜度：<run_id / capture_id / browser_profile / state_reset / old-vs-new / verified-vs-unverified>
收尾清理：<删除哪些临时/历史/废代码/废注释,保留哪些证据,加密算法细节图路径>
质量闸门：<Validation Ledger / Refusal Ledger / Human Review Ledger / Monitoring Ledger / Error Correction Ledger>
```

输出后等用户确认或修正，再进入执行。

## 0.1 交付 / 经验 / SKILLS 三层隔离（强制）

任何实战任务必须先区分三类产物,不能互相替代:

| 层 | 允许内容 | 禁止内容 |
|---|---|---|
| 项目交付 | 用户可运行的单一交付文件、交付包、adapter 包、runbook、接口服务代码 | 作为 SKILLS 正向样本、直接写入 skill/evals、混入经验库正文 |
| 项目经验库 | 真实接口测试确认后的事实摘要、失败模式、证据索引、图谱、影响回归、监控要求 | 放置项目交付代码、demo 服务、一次性脚本、未清洗 HAR/token payload、研发期猜测 |
| SKILLS 能力层 | 经过成功经验库验证的通用规则、负例 eval、治理门槛、抽象模板 | 直接吸收未成功项目代码、blocked adapter、单站点临时代码 |

硬规则:

- 项目交付是项目产物,不是 SKILLS 能力本身。
- 项目经验是交付成功或明确失败后的精炼沉淀摘要,不是交付项目代码仓库。
- 研发/探索阶段只能写本地临时项目库、抓包目录或交付目录;正式经验库必须等真实接口测试后再写入。
- 正式经验库必须区分 `已完成确认`、`未完成确认`、`踩坑点`、`下次不再重复路径` 和 `剩余风险`。
- 交付项目代码不准参与 SKILLS 评分、泛化或正向能力声明。
- 只有 `success_browserless_verified` 或同等级真实成功状态的经验库条目,才允许作为 SKILLS 正向经验参与评分和能力升级。
- `blocked_by_protection`、`blocked_by_manual_challenge`、`partial_adapter_only`、`negative_baseline_only` 只能进入 known-failures、test-log、refusal/human-review ledger 或负例 eval,不能写成可复用成功经验。
- 对 CAPTCHA/WAF/风控,只有授权测试中取得真实 token/state 且最终业务接口 repeat_verified 后端接受,才可声明自动化能力;否则只能写 `human_reviewed`、`blocked`、`incomplete` 或负例。

一体化路径:

| 阶段 | 路径 | 关键输出 | 准入规则 |
|---|---|---|---|
| 规划 | `99-SKILLS治理/06-网页逆向标准规划.md` | 目标、scope、证据口径、风险红线 | 用户确认或修正后执行 |
| 研发探索 | 本地临时项目库 / `交付项目/<domain>/` / 抓包目录 | HAR、完整 req/resp、临时脚本、失败尝试 | 本地保留,不进 GitHub |
| 真实测试确认 | run ledger / validation ledger / replay-diff 输出 | 已完成确认、未完成确认、阻塞原因 | 必须有真实接口证据 |
| 正式经验 | `站点经验库/<domain>/`、`逆向工程经验库/domains/<domain>/`、`验证码经验库/domains/<domain>/` | 脱敏摘要、踩坑点、下次排查顺序 | 真实 domain 默认本地保留 |
| 知识图谱 | `站点经验库/<domain>/knowledge-graph.md` | 节点、边、事实等级、证据、回归引用 | 改端点/字段/状态/保护/实现/eval 必写 |
| 影响回归 | `站点经验库/<domain>/impact-regression.md` | 影响面、必跑回归、数据校验、漂移风险 | 无 Impact Record 不得声明完成 |
| SKILLS 升级 | SKILL / references / evals / governance | 通用规则、负例门槛、评分准入 | 只有成功经验可正向升级 |
| 收尾 | cleanup ledger / encryption algorithm graph | 清理账本、算法图、遗留风险 | 完成前必须输出 |

## 1. 分类阶段（强制）

调用 `02-新网站接入分类.md` 的一级分类表，给目标网站打标签。一个网站可能命中多个标签（典型：纯 HTTP 阶段 + JS 加密阶段 + WAF 阶段并存）。

**分类决定调用哪个一级入口**：

| 命中标签 | 一级入口 |
|---|---|
| 纯 HTTP / JS 加密 / 补环境 | `reverse-js-crawler` |
| WAF / 风控 / 84盾 / Incapsula | `imperva-waf-reese84` |
| 验证码 / reCAPTCHA / hCaptcha / Turnstile / 滑块 / 点选 | `captcha-service-delivery` |
| 接口已稳定，要做 adapter | `site-api-adapter` |
| 用户要求纯接口全链路、FastAPI 接口测试交付、或接口成功后接入本地基础框架 | `website-314-api-delivery`（总控） |

如果用户的目标是"纯接口实现 + FastAPI 接口测试交付 / 服务化"，从 `website-314-api-delivery` 开始；否则直接用对应的二级 skill。314 只有在 Python/FastAPI 接口全部成功且用户确认接入后，才作为本地基础框架分支执行。

## 1.5 AI 自理能力（强制）

进入实现前必须建立三类自理约束：

| 类别 | 规约 | 最低要求 |
|---|---|---|
| 事实证据 | `11-AI事实证据规约.md` | 关键结论标 observed / derived / assumed / unverified |
| 反泛化收敛 | `12-反泛化与任务收敛规约.md` | 写明 in_scope / out_of_scope / unknown |
| 并发会话隔离 | `13-并发指纹与会话隔离规约.md` | 未跑并发阶梯不得声称支持并发 |
| 图谱行程关联 | `14-知识图谱行程与关联规约.md` | 端点/字段/状态/保护/实现/eval 更新必须写节点和边 |
| AI 变更回归 | `15-AI变更风险与回归校验规约.md` | 每次改动必须写 Impact Record 和必跑回归 |
| 实战复测新鲜度 | `16-实战复测与证据新鲜度规约.md` | run 账本、浏览器清空、抓包、多轮对照、旧新数据隔离、经验写回 |
| 交付收尾清理 | `17-交付收尾清理与加密算法图谱规约.md` | 验证后清理临时/历史/废代码/废注释,补整体加密算法图 |
| 证据验证拒答人审监控 | `18-证据验证拒答人工复核与监控规约.md` | 证据/验证/拒答/人工复核/监控/错误纠正/历史遗留必须入账 |

所有阶段的交付都要能回答：

1. 这个结论的证据是什么？
2. 这个结论覆盖哪个 market/stage/session？
3. 这个结论有没有被泛化到未验证范围？
4. 如果涉及批量/并发/指纹/session/cache，隔离证据在哪里？
5. 这次新增或修改了哪些图谱节点/边？
6. 这次改动的直接影响、下游影响和必跑回归是什么？
7. 这次证据是 fresh 还是 stale？是否区分 run_id / clean_unverified / verified / repeat_verified？
8. 验证完成后哪些临时文件、历史数据、废代码、废注释已清理？加密算法细节图在哪里？
9. 有无拒答/人工复核/错误纠正/监控项？对应 ledger 在哪里？

## 2. 六阶段执行模板

### 阶段 A：侦察（reconnaissance）

| 调用 | 产出 |
|---|---|
| `reverse-js-crawler` workflow 第 1 步 | 页面路由表、官方入口、可见 API 列表 |
| Chrome DevTools（MCP js-reverse 优先） | network 日志、cookie 列表、headers 模板 |

侦察前先读 `逆向工程经验库/domains/<domain>/reverse-memory.md`。没有该文件时,先从 `逆向工程经验库/_templates/reverse-memory.md` 建立空记录,再分配新的 `run_id`。

**禁止**：不允许跳过侦察直接猜 API 名。如果页面没有真实业务能力，**报告"页面无能力"，不能伪造成功**。
所有侦察结论必须按 `11-AI事实证据规约.md` 标注等级和证据。

### 阶段 B：入口定位（entry）

| 情境 | 调用 |
|---|---|
| 找 sign/token/x-sign 生成位置 | `find-crypto-entry` |
| 看不懂的混淆 JS | `ast-deobfuscate` |
| 命中 84盾 / Incapsula / x-d-token / reese84 cookie | 切到 `imperva-waf-reese84`，不要硬抠 |
| 命中验证码 / reCAPTCHA / hCaptcha / Turnstile / 滑块 / 点选 | 切到 `captcha-service-delivery`,先读验证码经验库,再做 clean_unverified / verified / repeat_verified 三组抓包 |

### 阶段 C：还原（restore）

| 情境 | 调用 |
|---|---|
| 加密函数依赖 window/document/navigator/canvas | `env-patch`（补环境） |
| 算法是纯 JS，能直接 port | 在 Node 复现算法 |
| 算法是 WASM | 在浏览器或 Node WASM runtime 中最小化调用,记录导入表和输入输出样本 |
| 算法不在 Web/H5 页面或浏览器 JS 内 | 记录不适用边界,本仓库停止处理 |

### 阶段 D：复现 + 验证（reproduce + verify）

| 步骤 | 标准 |
|---|---|
| 单接口复现 | 同一参数同一身份，能稳定拿到与浏览器一致的响应 |
| 链路串接 | 查询 → 加车 → 生单 → 支付 全链路通过 |
| WAF 接受度 | 至少 3 个不同 IP / 3 次重启浏览器后仍能通过 |
| 失败路径 | 主动测试 400/403/503，区分业务错/路由错/加密错/WAF |
| 并发声明 | 必须有 `13-并发指纹与会话隔离规约.md` 的阶梯记录和隔离说明 |

支付阶段强制：sandbox / dry-run / 支付前置接口优先，真实扣款必须用户明示授权。

### 阶段 E：沉淀（persist）

任务结束**强制**走以下五步，否则不算完成。沉淀前先判定交付状态;只有真实接口测试确认后的成功交付才能进入正向经验库,失败或阻塞只能进入失败/复核/监控账本。

1. 写 `站点经验库/<domain>/known-failures.md`（失败模式：symptom / stage / market / currency / status / marker / root cause / correct handling）
2. 写 `站点经验库/<domain>/test-log-lessons.md`（这次测试学到什么）
3. 写 `站点经验库/<domain>/change-log.md`（变更记录）
4. 调用 `5-沉淀工具层/site-api-adapter` 产出 adapter.yaml / schema.json（仅当接口已稳定且真实成功）
5. 调用 `skills-evaluation-governance`：只有成功经验库可作为正向样本;失败/blocked/adapter-only 只能补负例 eval 或门槛规则
6. 更新 `站点经验库/<domain>/knowledge-graph.md`：记录本次新增/修改的节点和边
7. 更新 `站点经验库/<domain>/impact-regression.md`：记录影响面、必跑回归、数据校验和剩余风险
8. 更新 `逆向工程经验库/domains/<domain>/reverse-memory.md`：记录本轮 run/capture/replay、old-vs-new diff、工具失败样本和下次排查顺序

知识图谱最低落点:

| 任务事实 | 必写节点 | 必写边 |
|---|---|---|
| 发现真实业务接口 | `endpoint` | `stage -> endpoint` |
| 接口依赖请求字段/header/cookie/body | `request-field`、必要时 `state` | `endpoint -> request-field`、`request-field -> state` |
| 交付字段来自响应 | `response-field` | `endpoint -> response-field` |
| 命中 WAF/验证码/风控 | `protection` | `endpoint -> protection` |
| 写了 adapter/脚本/服务代码 | `implementation` | `implementation -> endpoint` |
| 新增负例或回归评测 | `eval` | `eval -> node/edge` |

沉淀状态必须显式写入:

```yaml
delivery_status: success_browserless_verified | blocked_by_protection | blocked_by_auth | blocked_by_manual_challenge | partial_adapter_only | negative_baseline_only | unverified
skills_participation: positive_allowed | negative_eval_only | memory_only | prohibited
project_delivery_artifact: <单一交付文件/交付包路径>
experience_memory_path: <沉淀摘要路径>
```

当 `delivery_status` 不是成功态时,`skills_participation` 不能是 `positive_allowed`。

研发阶段的中间材料不进入正式经验库:完整 HAR、完整 req/resp、cookie、token、订单参数、账号态、浏览器 profile、临时脚本和未验证推断只能留在本地临时项目库。正式经验库只写脱敏后的可复用操作结论和状态确认。

### 阶段 F：一致性验证（consistency）

**没有这步,前面五个阶段都只是"看起来完成"**。详见 `99-SKILLS治理/07-一致性验证规约.md`。

| 步骤 | 标准 |
|---|---|
| 录制 fixtures | 用 `tools/recorder/cloak_recorder.py` 或 `har_to_fixtures.py` 在浏览器抓真实业务流程,落 `站点经验库/<domain>/fixtures/snapshots/` |
| 人工 review meta.yaml | 填 `endpoint` / `category` / `sensitive` / `tolerance`,标支付/PII |
| 重放 + diff | `snapshot_replay.py --target <adapter-url>` + `consistency_report.py --threshold 0.90` |
| 一致率 ≥ 90% | < 90% 必须修后重跑,不算交付 |
| 失败模式回写 | 一致率长期 < 90% 的根本原因写进 `known-failures.md` + 加 eval |

**红线**:`category` 不允许 `payment` / `order-create`,fixtures 不录支付链路。

CI 每周二自动重放,< 90% 自动开 issue。

### 阶段 G：交付收尾（cleanup + crypto graph）

详见 `99-SKILLS治理/17-交付收尾清理与加密算法图谱规约.md`。

| 步骤 | 标准 |
|---|---|
| 清理临时测试文件 | 已被正式 eval/fixture/script 替代的临时文件删除 |
| 清理历史数据 | 旧评分、旧 debug dump、过期中间产物已迁移必要证据后删除 |
| 清理废代码/废注释 | 无引用代码、过期 TODO、与当前事实不符注释删除或修正 |
| 证据保留 | HAR/fixture/script hash/失败样本若仍是唯一证据,先迁移到 memory/meta 再处理 |
| 加密算法细节图 | 涉及 sign/token/加密时写 `逆向工程经验库/domains/<domain>/encryption-algorithm-graph.md` |
| 清理账本 | 输出 Cleanup Ledger: removed / kept_as_evidence / migrated_to_memory / remaining_reason |

没有阶段 G,不能声明"完整交付"。

### 阶段 H：质量闸门（evidence + validation + review + monitoring）

详见 `99-SKILLS治理/18-证据验证拒答人工复核与监控规约.md`。

| 步骤 | 标准 |
|---|---|
| 证据门槛 | endpoint/字段/算法/WAF/并发/清理/错误修复都有可追溯证据 |
| 验证账本 | 输出 Validation Ledger,列 target/evidence/command/expected/observed/pass_fail |
| 拒答账本 | 越界请求写 Refusal Ledger,拒绝不安全部分并给安全替代 |
| 人工复核 | 支付/登录态/PII/删除唯一证据/生产影响/证据冲突进入 Human Review Ledger |
| 错误纠正 | AI 或工具出错后更新 memory/known-failures/eval/图谱/影响回归,删除或隔离错误代码 |
| 目标纠偏 | 偏离最初目标或上下文发散时写 Goal Recenter Note,压缩上下文并回到 in_scope |
| 监控 | 输出 Monitoring Ledger,覆盖 replay/schema/字段/评分/证据过期/历史遗留 |

没有阶段 H,不能声称"稳定"、"长期可用"或"治理闭环完成"。

## 3. 进化闭环（每次任务后必跑）

任务结束后回答 6 个问题：

1. 有没有新触发词？→ 更新对应 skill 的 description
2. 有没有新失败类型？→ `known-failures.md` + eval
3. 有没有新分类规则？→ `02-新网站接入分类.md` 加行
4. 有没有新加解密或反爬模式？→ `references/` 新增章节
5. 有没有应该加入 eval 的场景？→ `evals/`
6. 需不需要升版本号？→ 看 `03-测试评分漂移.md`
7. 有没有错误纠正、人工复核或监控项？→ 写 ledger + eval + monitoring plan

## 4. 走查示例：航司类加购 + 支付站点

通用 walkthrough，验证规划框架是否覆盖典型多分支多模块站点。

**触发**：「实现 example-airline.com 的某加购模块 + 支付，加解密全部实现，先做 FastAPI 接口测试交付，成功后确认是否接入 314 框架」

**分类**：
- 纯 HTTP ✓（booking / ancillary API）
- JS 加密 ✓（请求 waf signals）
- 补环境 ✓（站点模块化的 waf solver JS）
- WAF ✓（第三方支付网关 + reCAPTCHA Enterprise）
- FastAPI 接口测试交付 ✓
- 本地基础框架接入待确认（314 / flight_cwl_common_314 只是可选分支）

**阶段化规划**：
```
1. 侦察   skill: reverse-js-crawler   产出: 区分加购的各功能模块各自链路、区分多个支付网关分支
2. 入口   skill: find-crypto-entry    产出: waf signals 生成函数位置
3. 还原   skill: env-patch            产出: 各模块 solver JS 在 Node 跑通
4. 复现   skill: reverse-js-crawler   产出: 全链路接口脚本跑通
5. 沉淀   skill: site-api-adapter + skills-evaluation-governance
6. 一致性 skill: tools/replayer/*     产出: fixtures replay + diff report
```

**逆向经验库**：在 `逆向工程经验库/domains/<domain>/reverse-memory.md` 记录 run/capture/replay、旧新数据 diff、工具失败样本和下次排查顺序。

**站点经验库**：在 `站点经验库/<domain>/` 建立完整 7 文件结构（任务结束用 `tools/sync_site_memory.py` 半自动同步 project memory）。

**WAF 预判**：reCAPTCHA Enterprise 出现在 payment-methods 接口时，**mcp Chrome stealth 治标**，真扣款走真实浏览器或专用工具。

**收尾**：完成一致性验证后,写 cleanup ledger;涉及 waf signals/sign/token 时,补整体加密算法细节图。

**红线（典型）**：
- 站点不同业务模块（加购 / 选座 / 险种 / 行程附加）通常是**独立链路**，改一个不影响另一个；共享层有需求改子类 override
- 同一支付往往存在多个网关分支（issue_web / checkout / payment 等），样本必须按分支隔离
- 真实扣款一律不在自动化测试环境跑

## 6. 失败回退路径

| 卡点 | 回退 |
|---|---|
| 找不到加密入口 | `find-crypto-entry` 失败 → `ast-deobfuscate` 先把代码看懂 |
| 补环境补不动 | `env-patch` 失败 → 退回浏览器内执行（mcp / 真实浏览器） |
| WAF token 被拒 | 切 `imperva-waf-reese84` → 实在不行用真实浏览器 + 身份缓存 |
| 算法不在 Web/H5 页面或浏览器 JS 内 | 记录不适用边界,本仓库停止处理 |
| 接口能通但业务错 | 不要硬补参数，回阶段 A 检查路由 / market / locale / currency |

## 7. 输出规范

每次规划报告给用户时，遵循 `karpathy-guidelines`：

- 不默默假设：所有不确定的判断显式标"未验证"
- 最小改动：先复现一个最小请求成功，再扩链路
- 精准边界：明确这次任务不做的事（"本次不做 sandbox 真实扣款"）
- 目标驱动验证：每阶段给出"成功标准"，否则不算结束
- 事实分级：observed / derived / assumed / unverified 不混写
- 反泛化：单接口、单 market、单 session 成功不等于全链路成功
- 并发边界：未跑阶梯验证不声称支持并发或高并发
- 图谱更新：改端点/字段/状态/保护/实现/eval 必须更新节点和边
- 影响回归：改动后必须列 direct_impact、downstream_impact、required_regression
- 证据新鲜度：旧 HAR/旧 token/旧 scriptId/旧浏览器 profile 未复验前一律 stale
- 经验吸收：开始前读 reverse-memory，结束前写 run/capture/replay 和 old-vs-new diff
- 收尾清理：验证后删除临时测试文件、历史输出、废代码和废注释,保留证据必须说明原因
- 加密算法图：涉及 sign/token/加密时,必须交付整体算法细节图
- 拒答边界：无授权、无证据、绕过/伪造、删除唯一证据等请求必须拒绝或收缩范围
- 人工复核：高风险、不可逆、生产影响、证据冲突和跨范围扩展必须先人审
- 错误纠正：AI/工具犯错后必须更新 memory/eval/图谱/影响回归,删除或隔离错误代码
- 监控闭环：长期稳定声明必须有 replay/schema/字段/评分/证据过期监控
