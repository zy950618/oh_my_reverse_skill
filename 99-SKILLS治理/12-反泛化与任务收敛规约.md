# 反泛化与任务收敛规约

本规约解决 AI 在逆向任务中最容易犯的第二类错误：以点概面、发散扩写、把一个局部成功说成全局成功。

逆向任务必须围绕当前用户目标、当前 domain、当前 market、当前 stage 收敛。任何跨 market、跨账号、跨会话、跨业务阶段、跨时间窗口的结论，都必须单独验证。

## 不允许泛化的边界

| 局部事实 | 不能直接泛化为 |
|---|---|
| search 接口成功 | cart/order/payment 成功 |
| 一个 market 成功 | 所有 market 成功 |
| 一个 locale 成功 | 所有语言/币种成功 |
| 一个账号成功 | 所有账号/游客态成功 |
| 一个浏览器 session 成功 | 可并发、可长期复用 |
| token 生成成功 | 业务 API 接受 |
| 单次请求 200 | 稳定可交付 |
| 本地 replay 通过 | 线上服务高可用 |

## Scope Ledger

每个任务维护一个简短范围账本：

```text
in_scope:
  domain:
  market:
  locale:
  currency:
  stage:
  auth_state:
  target_api:
out_of_scope:
  - <明确不做的阶段/市场/账号/支付动作>
unknown:
  - <尚未验证但可能影响结论的变量>
```

没有进入 `in_scope` 的内容，最终答复不能说“已支持”。

## 阶段收敛规则

1. 先最小可行请求，再扩链路。
2. 先单线程稳定，再谈批量/并发。
3. 先当前 market，再扩其他 market。
4. 先公开/非破坏性接口，再碰登录态或敏感动作。
5. 每扩一个维度，都要新增证据和失败记录。

## 反发散规则

- 不主动扩展到用户没要求的业务模块。
- 不把“顺便支持所有站点”写进交付。
- 不做 speculative abstraction。
- 不因为看到 WAF/指纹就切到全套风控方案；先判断当前阶段是否真的被保护。
- 不因为发现一个失败就硬编码站点专属 workaround，必须先判断是否是通用失败类型。

## 最终答复必须说明

- 本次验证覆盖了哪些 market/stage/session。
- 哪些维度没有覆盖。
- 是否只验证了单次、连续、多次重启、还是并发。
- 是否只验证了 token 生成，还是业务 API 接受。

## 回归 eval 建议

每个业务入口 skill 至少包含这些负例/回归例：

- 单接口成功后，模型不得声称全链路成功。
- 单 market 成功后，模型不得声称全 market 支持。
- token 生成成功后，模型不得声称 WAF 接受。
- 用户只要定位入口时，模型不得擅自实现完整采集工程。
