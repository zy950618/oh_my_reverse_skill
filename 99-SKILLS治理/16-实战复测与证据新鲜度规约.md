# 实战复测与证据新鲜度规约

本规约解决一个核心问题:AI 每次像重新开荒,分不清旧数据和新数据,把缓存、旧 token、旧 HAR、旧脚本 hash 当成当前事实。

适用范围是所有 Web/H5 逆向工程,不只验证码。凡是侦察、入口定位、解混淆、补环境、接口复现、WAF/风控、验证码、adapter 沉淀或一致性回放涉及旧证据复用,都按本规约执行。

## 触发场景

涉及以下任一情况必须执行本规约:

- 浏览器抓包、HAR、Network 事件、DevTools trace
- cookie、localStorage、sessionStorage、cache、service worker
- sign/token/riskToken/captcha token/session state
- WAF/验证码/滑块/点选/Turnstile/reCAPTCHA/hCaptcha
- 多次刷新、多轮对照、并发、复测、回归验证

## 证据新鲜度字段

任何关键证据必须能回答:

| 字段 | 要求 |
|---|---|
| `capture_id` | 本次抓包唯一 ID,不能只写"刚才抓的" |
| `captured_at` | ISO 时间戳 |
| `browser_profile_id` | 浏览器 profile 或用户数据目录标识 |
| `state_reset` | 是否清 cookie/storage/cache/service worker |
| `network_log_id` | HAR 文件、DevTools request id 或事件序号 |
| `script_hash` | JS 文件 URL/hash/etag/mtime |
| `auth_state` | anonymous/login/verified/unverified |
| `session_id_hint` | 不暴露敏感值的会话标识摘要 |
| `source_freshness` | fresh/stale/unknown |
| `run_lineage` | 本轮抓包/脚本/回放来自哪一次 run,是否继承旧资料 |

缺少这些字段时,不能把结论标成 `observed`,最多是 `unverified`。

## 逆向 run 账本

每次重新开浏览器、重新抓包、重新下载脚本、重新回放接口,都要分配新的 `run_id`:

```yaml
run_id:
started_at:
operator:
domain:
stage:
auth_state:
browser_profile_id:
state_reset:
source_memory_read:
capture_outputs:
script_hashes:
replay_outputs:
freshness_decision:
```

同一轮里的 HAR、DevTools request id、script hash、断点栈、回放结果必须能关联到同一个 `run_id`。跨 run 复用任何字段都要写 `old-vs-new diff`。

## 干净复测流程

最小实战复测必须跑三组:

1. `clean_unverified`:清空 cookie/storage/cache/service worker 后抓未验证状态。
2. `verified`:人工或授权流程完成验证后抓已验证状态。
3. `repeat_verified`:重新打开浏览器或刷新会话后再抓一次已验证状态。

每组都要记录:

- 请求 URL、method、headers/body 摘要
- 响应 status、业务 code、关键 JSON Pointer
- cookie/storage 写入变化
- token 字段、TTL、绑定关系
- 相关脚本 URL/hash

## 旧新数据隔离

禁止以下做法:

- 用昨天 HAR 的 token 解释今天接口。
- 用旧 scriptId 报当前函数位置。
- 清空浏览器后仍沿用旧 localStorage 结论。
- 把人工验证后的 cookie 当成未验证状态下的自然结果。
- 把单次刷新成功当成稳定成功。

如果必须复用旧经验,必须写:

```text
reused_evidence:
  source:
  captured_at:
  still_valid_because:
  revalidated_by:
```

## 多轮对照矩阵

| 对照 | 目的 |
|---|---|
| unverified vs verified | 找验证服务对业务 API 的真实影响 |
| first verified vs repeat verified | 判断 token/session 是否稳定 |
| old capture vs new capture | 防止旧数据污染 |
| same session vs new session | 判断会话绑定 |
| same market vs new market | 防止以点概面 |
| single request vs ladder | 判断并发/频率边界 |

## 经验库复用顺序

每次实战任务必须先读经验库再动浏览器:

1. `逆向工程经验库/domains/<domain>/reverse-memory.md`
2. `站点经验库/<domain>/known-failures.md`
3. `站点经验库/<domain>/test-log-lessons.md`
4. `站点经验库/<domain>/knowledge-graph.md`
5. `站点经验库/<domain>/impact-regression.md`
6. 如涉及验证码,再读 `验证码经验库/providers/<provider>.md` 和 `验证码经验库/domains/<domain>/captcha-memory.md`

没有读旧经验就开始抓包,属于流程缺陷。

## 经验沉淀成熟度

实战经验持续沉淀,但不能在研发阶段直接污染正式经验库。按三层处理:

| 阶段 | 保存位置 | 允许内容 | 禁止升级 |
|---|---|---|---|
| 研发/探索 | 本地临时项目库、抓包目录、交付目录 | HAR、完整请求响应、调试脚本、失败尝试、未验证假设 | 不进入 GitHub,不进入正向 SKILLS |
| 测试确认 | 本地 run 账本、validation ledger、diff 报告 | 已完成确认、未完成确认、真实接口测试结果、阻塞原因 | 无后端接受证据不得写成功 |
| 正式经验库 | `站点经验库/`、`逆向工程经验库/`、`验证码经验库/` 的脱敏摘要 | 可复用操作结论、失败模式、下次排查顺序、影响回归、算法图谱摘要 | 不写真实 token/cookie/订单/完整 payload |

进入正式经验库必须满足:

- 有真实接口测试或明确的失败/阻塞证据。
- 明确 `已完成确认` 或 `未完成确认`。
- 记录踩坑点和下次不必重复处理的错误路径。
- 只保留脱敏摘要、字段含义、JSON Pointer、状态机、验证命令和回归要求。
- 对 WAF/验证码/风控,只有最终业务 API 后端接受且 repeat_verified 通过,才能作为成功经验;否则只能作为 blocked/negative/human-review 经验。

## 基础逆向经验吸收

每次任务结束必须把以下内容写回 `逆向工程经验库/domains/<domain>/reverse-memory.md` 或对应模板文件:

- 本轮 run/capture/replay 的最小可复现路径。
- 旧数据被复验通过、复验失败或判定 stale 的原因。
- 浏览器清空前后差异:cookie/storage/cache/service worker/script hash/request id。
- 入口定位的真实调用链和错误调用链。
- 解混淆、补环境、回放过程中踩到的工具失败样本。
- 数据偏移:JSON Pointer、字段类型、数组顺序、分页/market/session 差异。
- 下次必须先看的证据和必须先排除的旧结论。

## 交付硬门槛

最终报告必须包含:

```text
Fresh Evidence Table:
  run_id / capture_id / captured_at / profile / state_reset / auth_state / request_id / source_freshness

Old-vs-New Diff:
  old evidence:
  new evidence:
  still valid:
  invalidated:

Verified-vs-Unverified Diff:
  changed request fields:
  changed response pointers:
  changed cookies/storage:
  unlocked endpoints:

Retest Matrix:
  clean_unverified:
  verified:
  repeat_verified:

Reuse Decision:
  reused prior experience:
  new experience added:
```

缺少以上任一项,不得声明"稳定"、"可复用"、"支持并发"或"已完成"。
