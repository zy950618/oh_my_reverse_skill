# 知识图谱行程与关联规约

本规约只解决一个问题：每次更新后，当前站点的节点、边、来源、下游关联必须被记录清楚。

它不评价 AI 行为，也不定义回测策略；AI 风险和回归校验见 `15-AI变更风险与回归校验规约.md`。

## 必须维护的图谱文件

每个 domain 都应维护：

- `knowledge-graph.md`: 当前节点和边。
- `impact-regression.md`: 每次更新影响了哪些节点/边，以及对应回测记录。

## 节点

知识图谱只记录当前 Web/H5 仓库范围内的事实节点。

| 节点类型 | 示例 | 必填证据 |
|---|---|---|
| domain | `example.com` | 官方入口 URL |
| market | `TH / en / THB` | 页面 route 或接口参数 |
| stage | `search / cart / order / config` | 页面行为或业务链路证据 |
| endpoint | `POST /api/search` | HAR / Network / req snapshot |
| request-field | `headers.x-sign` | 调用栈 / hook / bundle 位置 |
| response-field | `/data/flights/0/price` | JSON Pointer / fixture |
| state | `booking_session_id` | cookie/storage/response source |
| protection | `waf-cookie / tls-fingerprint / rate-limit` | 响应码、响应头、cookie、body marker |
| implementation | `adapter route / mapper / crypto fn` | 文件路径、函数名、测试 |
| eval | `boundary-overgeneralization` | eval 文件路径 |

节点状态只能是：

- `observed`: 当前任务直接观察到。
- `derived`: 有证据推导。
- `assumed`: 明示假设。
- `unverified`: 未验证。

## 边

边必须描述“为什么关联”，不能只写两个名字。

| 边 | 含义 |
|---|---|
| `domain -> market` | 哪些 market 已知存在 |
| `market -> stage` | 哪些 stage 在该 market 验证过 |
| `stage -> endpoint` | 该阶段调用哪些真实端点 |
| `endpoint -> request-field` | 请求依赖哪些字段 |
| `endpoint -> response-field` | 交付依赖哪些响应字段 |
| `request-field -> state` | 字段来源于哪个状态 |
| `endpoint -> protection` | 端点是否受保护或频控 |
| `implementation -> endpoint` | 哪段代码实现哪个端点 |
| `eval -> node/edge` | 哪个 eval 覆盖哪个节点或边 |

## 更新规则

每次更新以下内容，必须同步更新图谱：

- 新增或修改 endpoint。
- 新增或修改请求头、cookie、query/body 字段、storage 状态。
- 新增或修改 response mapper、schema、JSON Pointer。
- 新增或修改 protection 观察。
- 新增或修改 adapter route、实现函数、eval。
- 新增或修改 market / locale / currency / stage。

## 禁止项

- 不允许只改实现，不更新图谱。
- 不允许只写“影响不大”，必须列出直接关联和下游关联。
- 不允许把一次成功写成所有边都通过。
- 不允许新增没有证据的 observed 节点。
- 不允许跨 market/stage/session 复用边，除非单独验证。
