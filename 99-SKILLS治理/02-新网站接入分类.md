# 新网站接入分类

当用户丢进来一个新网站时，先分类，不要直接写代码。

## 一级分类

| 类型 | 判断特征 | 进入 Skill |
|---|---|---|
| 纯 HTTP | 参数稳定、返回 JSON、无动态签名 | `reverse-js-crawler` |
| JS 加密 | `sign`、`token`、`x-sign`、`authKey`、混淆 JS | `reverse-js-crawler` |
| 补环境 | 依赖 `window/document/navigator/canvas/localStorage` | `reverse-js-crawler` |
| WAF/风控 | 403、HTML challenge、84盾、Incapsula、Reese84 | `imperva-waf-reese84` |
| 验证码/验证服务 | reCAPTCHA、hCaptcha、Turnstile、滑块、点选、sitekey、challenge、verify endpoint、captcha token | `captcha-service-delivery` |
| 接口化沉淀 | 已有结果，需要 adapter/schema/runbook | `site-api-adapter` |
| FastAPI 接口测试交付 | 用户要求新网站纯接口、HTTP API、可测试接口服务 | `website-314-api-delivery` |
| 本地基础框架接入 | FastAPI 接口全部成功后，用户确认接入 314 或其他本地基础框架 | `website-314-api-delivery` |
| 技能治理 | 评分、补 eval、漂移测试、版本维护 | `skills-evaluation-governance` |

## 二级分类：业务阶段

电商/航司/旅游类网站通常拆成：

- 查询
- 价格/库存
- 加车/选择报价
- 旅客/收件/联系人
- 生单/订单确认
- 支付初始化
- 支付提交
- 查询订单/出票/确认

每个阶段都要单独判断：

- host/domain 是否变化
- cookie 是否变化
- token 是否变化
- WAF 是否变化
- 验证码/验证服务是否变化
- 旧抓包/旧 token 是否仍然有效
- 成功标准是什么

## Thai Airways 类型任务

如果用户说：

```text
实现 https://www.thaiairways.com/ 查询、加车、生单、支付，
加解密全部实现，先提供 FastAPI 接口测试；全部成功后再确认是否接入 314 或其他本地基础框架
```

应当进入：

```text
website-314-api-delivery
  -> reverse-js-crawler
  -> imperva-waf-reese84
  -> site-api-adapter
  -> skills-evaluation-governance
```

不能只把它当普通采集脚本处理。
