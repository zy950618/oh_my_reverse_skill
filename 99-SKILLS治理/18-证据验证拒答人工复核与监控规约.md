# 证据验证拒答人工复核与监控规约

本规约补齐五个交付缺口:

- 证据:关键结论必须绑定可追溯证据,不能只写判断。
- 验证:每个实现、字段、端点、算法和清理动作必须有验证路径。
- 人工复核:高风险、不可逆、证据冲突或生产影响动作必须进入人工复核。
- 监控:交付后要持续发现漂移、过期证据、错误代码和历史遗留。

它不替代 `11-AI事实证据规约.md` 到 `17-交付收尾清理与加密算法图谱规约.md`,而是把这些规约串成一个质量闸门。

## 证据门槛

每条重要结论至少绑定一种证据:

| 结论类型 | 最低证据 |
|---|---|
| endpoint 存在 | URL / method / status / HAR entry 或 DevTools request id |
| 请求字段有效 | 调用栈 / hook 记录 / bundle 位置 / replay 结果 |
| 响应字段可交付 | 最终成功 API 响应 + JSON Pointer |
| sign/token 算法 | 脚本 URL/hash + 函数链 + 输入输出样本 |
| WAF/验证码影响 | protected business API 接受度,不能只看 challenge/token endpoint |
| 并发稳定 | 并发阶梯记录 + session/cache 隔离证据 |
| 清理完成 | Cleanup Ledger + 保留证据迁移位置 |
| 错误已修复 | Error Correction Ledger + 回归命令输出 |

缺证据时,结论只能标 `assumed` 或 `unverified`。不能为了让报告好看把证据口径降级。

## 验证门槛

每次交付前必须回答:

```text
Validation Ledger:
  target:
  evidence:
  command_or_replay:
  expected:
  observed:
  pass_fail:
  remaining_gap:
```

同时必须回答交付状态和 SKILLS 参与资格:

```text
Delivery Status Ledger:
  project_delivery_artifact:
  delivery_status:
  real_success_evidence:
  adapter_only:
  blocked_reason:
  experience_memory_path:
  skills_participation: positive_allowed | negative_eval_only | memory_only | prohibited
```

规则:

- `adapter_only: true` 时不能声明真实成功。
- `blocked_reason` 非空时不能声明完整交付。
- 没有 `real_success_evidence` 时不能进入 `positive_allowed`。
- 项目交付代码不作为 SKILLS 输入;只有沉淀后的成功经验库摘要可以参与 SKILLS 正向能力升级。
- 对 CAPTCHA/WAF/风控,`real_success_evidence` 必须来自最终业务 API 的后端接受证据,不能用 challenge/config/provider endpoint 的 HTTP 200、403 challenge 页面或 UI 状态替代。

验证要求:

- 数据字段用 JSON Pointer 验证。
- 请求链路用 replay / fixture / HAR 对照验证。
- 代码改动用最小测试、回归脚本或评分脚本验证。
- 清理动作先迁移证据,再删除临时/错误/废弃文件。
- 验证失败不能只改措辞;必须写失败原因、修复动作和复测结果。

## 拒答边界

遇到以下请求必须拒答或收缩到安全范围:

- 真实扣款、真实下单、账号不可逆操作,但没有明确授权。
- 生成或提供 CAPTCHA 绕过、WAF 绕过、webdriver 隐藏、指纹伪造、代理轮换、clearance cookie 或风险 token。
- 猜测隐藏 endpoint、cookie、设备 ID、签名字段或访问控制参数。
- 删除唯一证据、覆盖用户改动、清空历史记录,但没有迁移证据或用户授权。
- 要求把 `assumed` / `unverified` 包装成 `observed`。
- 要求把辱骂性前缀、人格化称谓或非专业口头禅写入强制交付格式。

拒答格式:

```text
Refusal Ledger:
  request:
  refused_part:
  reason:
  safe_alternative:
  evidence_or_policy:
```

拒答后仍要继续推进允许范围内的工作,例如只做 observation-only 分析、证据映射、字段 parity、fixture replay 或人工复核清单。

## 人工复核门槛

以下动作必须暂停自动推进,输出 Human Review Ledger:

- 涉及支付、下单、登录态、PII、账号风控或真实业务损失。
- 删除 HAR、fixtures、日志、脚本 hash、错误样本等唯一证据。
- 把旧证据复用为当前 run 的事实。
- 修改生产 adapter、字段 mapper、加密算法或并发策略。
- 证据冲突:浏览器 UI、最终 API、replay、schema diff 任意两者不一致。
- 扩展到新 market / locale / currency / auth_state / session。
- 用户要求、事实证据或仓库规约冲突。
- 验证码 `verified` 或 `repeat_verified` 需要人工完成 challenge,而自动流程无法取得 token/state/backend acceptance。
- CAPTCHA/WAF/风控声称机器自动完成,但缺少授权范围、token/state 证据、最终业务 API 后端接受和 repeat_verified 复测。

```text
Human Review Ledger:
  decision_needed:
  risk:
  options:
  recommended_option:
  evidence:
  rollback:
```

人工复核不是停止工作。能做的低风险工作继续做:读 memory、整理证据、补 eval、写影响面、准备回归命令。

### 自动化与人工参与边界

自动化目标是减少重复操作和实现接口验证,不是把无证据的保护绕过写成能力。每次 CAPTCHA/WAF/风控任务必须标注:

```yaml
verification_mode: browser_automated_verified | human_reviewed_verified | blocked_by_manual_challenge | blocked_by_protection | unverified
backend_acceptance: <最终业务 API JSON Pointer / status / code>
repeat_verified: true | false
authorization_scope: <用户授权范围>
```

规则:

- `browser_automated_verified` 只允许在授权测试中使用,且必须有 token/state、最终业务 API 后端接受和 repeat_verified 证据。
- `human_reviewed_verified` 必须明确人工只完成可见 challenge,不执行越界业务动作。
- `blocked_by_manual_challenge` 可以沉淀为踩坑点或负例,不能写成"已支持自动通过"。
- 未取得后端接受证据时,不能把 Turnstile managed challenge、Akamai challenge、Cloudflare challenge 或任意 403/挑战页当成功。

### 验证码人工接入协议

当人工 challenge 阻塞 `verified` 或 `repeat_verified` 时,不能只写"需要人工"。必须输出可执行协议:

```text
Human Review Protocol:
  visible_browser: <可见浏览器启动方式或工具>
  browser_profile_id: <新 profile 或 user-data-dir>
  state_reset: <cookie/storage/cache/service worker 清理或记录>
  url: <精确 URL>
  variant: <widget/sitekey/mode>
  user_action: <用户只完成 challenge,不做越界动作>
  stop_after: <看到什么 UI/result 后停止>
  live_capture: <HAR/DevTools/webparity/MCP 监听方式>
  completion_criteria: <token 字段状态 + 后端 JSON Pointer 成功>
  repeat_flow: <新上下文/刷新后的 repeat_verified 流程>
  failure_path: <失败样本写入路径>
```

在人工接入完成前,最终状态只能是 `blocked` 或 `incomplete`,不能声明完成、稳定或长期可用。

## 错误纠正与历史遗留处理

AI 或工具出错后,不能只在聊天里道歉。必须进入错误纠正闭环:

1. 标记错误事实:把错误结论从 `observed` 降级为 `derived` / `assumed` / `unverified`。
2. 更新记忆:写入 `reverse-memory.md`、`known-failures.md`、`test-log-lessons.md` 或 eval。
3. 更新图谱:错误涉及 endpoint/字段/状态/保护/实现/eval 时,修正 `knowledge-graph.md` 和 `impact-regression.md`。
4. 删除或隔离错误代码:已被证明错误、无引用、无证据价值的代码删除;仍有证据价值的样本迁移到 failure fixture 或 memory。
5. 跑回归:用能复现错误的负例或 replay 验证修复。
6. 记录遗留:不能当场处理的历史问题进入 Legacy Debt Ledger,不能假装已经清理。

推荐路径:

```text
逆向工程经验库/domains/<domain>/error-correction-ledger.md
站点经验库/<domain>/known-failures.md
站点经验库/<domain>/impact-regression.md
```

没有真实 domain 时,先用 `逆向工程经验库/_templates/error-correction-ledger.md`。

## 目标纠偏与上下文压缩

当发现任务偏离最初目标、上下文过长或讨论开始发散时,不要继续扩大范围。先输出 Goal Recenter Note,再继续执行:

```text
Goal Recenter Note:
  original_goal:
  current_drift:
  discarded_context:
  kept_evidence:
  next_small_action:
  verification:
```

规则:

- 回到最初用户目标和当前 in_scope。
- 压缩上下文时保留证据、命令、文件路径、失败原因和未验证项。
- 丢弃无证据猜测、重复解释、无关分支和情绪化输出。
- 不把辱骂性前缀或固定称呼作为纠偏机制。
- 纠偏后继续执行可验证的下一步,不要停在口头反省。

## 监控门槛

交付后必须能监控这些信号:

| 信号 | 监控方式 |
|---|---|
| fixture 漂移 | `snapshot_replay.py` + `consistency_report.py` |
| schema 漂移 | `schema_alert.py` |
| 字段偏移 | `snapshot_diff.py` + JSON Pointer 清单 |
| skill 质量漂移 | `score_skills.py` + `ci_gate.py` |
| 证据过期 | fixture meta `expires_at` / run ledger `source_freshness` |
| 历史遗留 | Legacy Debt Ledger 定期清理 |
| 错误复发 | known-failures + 负例 eval |

```text
Monitoring Ledger:
  target:
  signal:
  cadence:
  command:
  threshold:
  owner:
  last_run:
  next_action:
```

没有监控记录时,不能声称"长期稳定"、"持续可用"或"已完成治理闭环"。

## 最终答复新增要求

最终答复除原有证据和收尾要求外,还必须说明:

- 哪些结论有证据。
- 哪些验证已跑,哪些没跑。
- 哪些请求被拒绝或收缩,安全替代是什么。
- 是否需要人工复核。
- 交付后监控什么,阈值是什么。
- 本轮是否修正了错误、删除了错误代码或留下历史遗留。
