# AI 事实证据规约

本规约解决 AI 在逆向任务中最容易犯的第一类错误：把猜测、推断、经验判断写成事实。

所有 Web/H5 逆向交付必须区分事实层级。没有证据的结论只能标为 `assumed` 或 `unverified`，不能写成已完成、已跑通、已定位、已证明。

## 结论分级

| 等级 | 名称 | 可以怎么说 | 禁止怎么说 |
|---|---|---|---|
| observed | 已观察事实 | 浏览器 Network 中观察到 `POST /api/search` 返回 200 | 这个接口一定是主接口 |
| derived | 有证据推导 | 调用栈显示 `buildSign()` 写入 `x-sign`，推断它是签名入口 | 已完全还原算法 |
| assumed | 明示假设 | 暂假设 market 只影响 currency，尚未跨 market 验证 | 所有 market 通用 |
| unverified | 未验证 | 未验证并发、登录态、长期稳定性 | 可高并发稳定运行 |

## 可接受证据

重要结论至少绑定一种证据：

- 请求证据：URL、method、status、request/response body、HAR entry。
- 代码证据：文件路径、函数名、行号、source map、bundle 片段。
- 运行时证据：XHR/fetch initiator、调用栈、hook 记录、WebSocket frame index。
- 数据证据：JSON Pointer、字段 diff、fixtures snapshot、replay report。
- 命令证据：命令行、退出码、关键输出。
- 站点记忆证据：`known-failures.md`、`market-matrix.md`、`route-decisions.md` 的记录。

## 交付格式

每个阶段输出都应包含：

```text
结论：<一句话>
等级：observed | derived | assumed | unverified
证据：<URL / 文件行号 / HAR entry / JSON Pointer / 命令输出>
限制：<没覆盖什么>
下一步验证：<如何把 assumed/unverified 推进到 observed>
```

最终答复必须包含：

- 做了什么。
- 哪些结论有直接证据。
- 哪些只是推断或假设。
- 哪些没有验证，不能被当成交付事实。

## 禁止项

- 不允许凭命名猜隐藏 endpoint。
- 不允许凭一次 200 声称“接口已稳定”。
- 不允许把 token/sign 生成成功写成业务 API 已接受。
- 不允许把本地脚本能跑写成线上可用。
- 不允许把评分高写成真实任务成功。
- 不允许把历史站点经验直接套到新站点，除非有当前站点证据。

## 自检问题

交付前逐条问：

1. 每个关键结论有没有证据？
2. 证据是否来自当前任务，而不是旧记忆？
3. 推断是否标为 `derived` 或 `assumed`？
4. 未验证项是否进入最终答复？
5. 有没有把“看起来像”写成“就是”？

## Direct Interface Positive Evidence Gate

Web/H5 reverse evidence is separated into four levels:

- `browser-only`: test evidence only. Browser capture, DOM text, UI/API parity, and browser repeat can help discover or compare behavior, but cannot enter `positive_allowed`.
- `browser-assisted discovery`: reconnaissance evidence only. It may identify candidate endpoints, fields, initiators, and UI mapping, but it is not interface success.
- `browser-captured replay`: boundary evidence unless sign/token/session material can be regenerated outside the browser. Replaying a copied browser request, cookie, token, storage value, profile, or HAR value cannot be positive capability.
- `direct interface accepted`: minimum positive evidence. The final API must be called by curl, Python requests/urllib, Node HTTP client, adapter, or another non-browser client and return accepted business JSON/data with JSON Pointer checks.

`positive_allowed` requires all of the following:

1. `final_api_endpoint_confirmed: true`.
2. `direct_interface_call.status: pass` with `browser_dependency: false`.
3. `uses_browser_profile: false`, `uses_live_storage: false`, and `uses_manual_cookie_or_token: false`.
4. 2xx response containing valid business JSON/data.
5. Non-empty JSON Pointer field acceptance from the direct response.
6. `repeat_direct_interface_call.status: pass` under the same non-browser constraints.
7. If sign/token is required, the generator must run outside the browser by V8/env patch, JS runtime, Node VM, wasm/crypto, native crypto, or adapter code. Captured browser token replay is not positive evidence.
8. If concurrency is claimed, require the 1/2/5/10 worker ladder, session/cache/token isolation, backend acceptance, failure rate, and stop condition evidence.

## Business Data Assertion Gate v2.1

Interface path success is not business success. HTTP 200 is not data correctness. Challenge or verify endpoint success is not final business API success. Worker PASS is not concurrency business consistency.

`positive_allowed` now requires four separate statuses:

```yaml
execution_status: REAL_EXECUTION_PASS
control_flow_status: CONTROL_FLOW_PASS
business_data_status: DATA_ASSERTION_PASS
capability_status: positive_allowed
```

Business data facts must be labeled at the same observed / derived / assumed / unverified levels as endpoint facts. `observed` business data requires a server-side business ledger or equivalent authoritative backend record.

Minimum evidence for `DATA_ASSERTION_PASS`:

- `business_data_assertions.server_ledger_path` exists.
- Positive assertions prove list/detail/submit data consistency.
- Negative assertions prove `expected_ledger_delta=0` and `actual_ledger_delta=0`.
- Concurrency assertions prove worker_1/2/5/10 success count equals business ledger delta.
- Order IDs are unique or follow a documented idempotency rule.
- No orphan order, wrong owner, cross-worker pollution, or unrecorded side effect exists.


## Phase 3.4 Runtime/Fingerprint Evidence Boundary

- JS Browser/Node/V8/PageRuntime parity is observed only for the named script fixture, input payload, run_id, and authorized scope.
- Runtime parity does not prove risk-token generation, WAF bypass, or third-party business API success.
- Fingerprint surface reports are observed/derived diagnostics. They must not be written as stealth, spoofing, or evasion capability.
- `positive_allowed` still requires final business API acceptance and `business_data_status=DATA_ASSERTION_PASS`.

## Phase 3.5 Longrun Evidence Boundary

- Source run_id: `run-20260630-041500-phase3-5-longrun`.
- Failure evidence must be cited from `public-range-evidence/longrun/phase3-5/<run_id>/failure-cases.json` or `issue-ledger.json`.
- Longrun CAPTCHA metrics, JS parity, fingerprint diagnostics, and localhost concurrency are observed only inside the configured local lab.
- `capability-decision.json` must keep every capability `memory_only` unless Phase 2.1 final business API and business-data assertions pass.
- Passing a longrun command does not prove third-party CAPTCHA/WAF/risk-control success.
