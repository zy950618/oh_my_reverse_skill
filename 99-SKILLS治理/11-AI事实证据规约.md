# AI 事实证据规约

本规约解决 AI 在逆向任务中最容易犯的第一类错误：把猜测、推断、经验判断写成事实。

所有 Web/H5 逆向交付必须区分事实层级。没有证据的结论只能标为 `assumed` 或 `unverified`，不能写成已完成、已跑通、已定位、已证明。

## 结论分级

| 等级 | 名称 | 可以怎么说 | 禁止怎么说 |
|---|---|---|---|
| observed | 已观察事实 | 浏览器 Network 中观察到 `POST /api/search` 返回 200 | 这个接口一定是主接口 |
| derived | 有证据推导 | 调用栈显示 `buildSign()` 写入 `x-sign`，推断它是签名入口 | 已完全还原算法 |
| assumed | 明示假设 | 暂假设 market 只影响 currency，尚未跨 market 验证 | 所有 market 通用 |
| unverified | 未验证 | 未验证并发、登录态、长期稳定性 | 可高并发稳定运行 |

## 可接受证据

重要结论至少绑定一种证据：

- 请求证据：URL、method、status、request/response body、HAR entry。
- 代码证据：文件路径、函数名、行号、source map、bundle 片段。
- 运行时证据：XHR/fetch initiator、调用栈、hook 记录、WebSocket frame index。
- 数据证据：JSON Pointer、字段 diff、fixtures snapshot、replay report。
- 命令证据：命令行、退出码、关键输出。
- 站点记忆证据：`known-failures.md`、`market-matrix.md`、`route-decisions.md` 的记录。

## 交付格式

每个阶段输出都应包含：

```text
结论：<一句话>
等级：observed | derived | assumed | unverified
证据：<URL / 文件行号 / HAR entry / JSON Pointer / 命令输出>
限制：<没覆盖什么>
下一步验证：<如何把 assumed/unverified 推进到 observed>
```

最终答复必须包含：

- 做了什么。
- 哪些结论有直接证据。
- 哪些只是推断或假设。
- 哪些没有验证，不能被当成交付事实。

## 禁止项

- 不允许凭命名猜隐藏 endpoint。
- 不允许凭一次 200 声称“接口已稳定”。
- 不允许把 token/sign 生成成功写成业务 API 已接受。
- 不允许把本地脚本能跑写成线上可用。
- 不允许把评分高写成真实任务成功。
- 不允许把历史站点经验直接套到新站点，除非有当前站点证据。

## 自检问题

交付前逐条问：

1. 每个关键结论有没有证据？
2. 证据是否来自当前任务，而不是旧记忆？
3. 推断是否标为 `derived` 或 `assumed`？
4. 未验证项是否进入最终答复？
5. 有没有把“看起来像”写成“就是”？
